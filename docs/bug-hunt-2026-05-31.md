## Confirmed Bugs

### BH-001 - Critical - Shop buy/sell actions are blocked by the overlay guard

Files:

- `src/js/input.js:2`
- `src/js/state.js:9`
- `src/js/shop.js:107`
- `src/js/shop.js:191`
- `src/js/shop.js:230`
- `autoplay_test.js:121`

Reproduction:

1. Serve the repo: `python -m http.server 8080 --bind 127.0.0.1`.
2. Open `http://127.0.0.1:8080/src/index.html` or `dungeon.html`.
3. Start a run and open a shop.
4. Try to buy an affordable item.
5. Switch to sell and try to sell any carried item.

Observed:

- The click/tap handler calls `buyItem()` or `sellItem()`, but player gold, inventory count, and item `sold` state do not change.
- This reproduces on desktop click and mobile portrait tap, in both `src/index.html` and `dungeon.html`.
- Targeted trace: `buy-enter overlay:true gold:100 sold:false carried:1` followed by `buy-exit gold:100 sold:false carried:1`.

Suspected root cause:

- `isOverlayOpen()` returns true when `#shop-overlay` is open.
- `canAct()` returns false whenever any overlay is open.
- Shop actions call `canAct()` from inside the shop overlay, so they are always rejected.
- Autoplay masks this by setting `window.canAct = () => true`.

Notes:

- Function-level shop logic worked when `canAct` was bypassed: potion, weapon, armor, upgrade purchase, insufficient-funds handling, sell item, and sell weaker gear all behaved plausibly.

### BH-002 - High - Lethal poison tick throws `showGameOver is not defined`

Files:

- `src/js/combat.js:244`
- `src/js/combat.js:254`

Reproduction:

Run this in the browser console after starting a run:

```js
G.enemies = [];
G.player.hp = 1;
G.player.poisonedTurns = 1;
advanceTurn();
```

Observed:

- Browser evaluation throws `showGameOver is not defined`.
- Reproduced in both `src/index.html` and `dungeon.html`.

Suspected root cause:

- Poison death path calls `showGameOver('Poison Gas')`, but the game only defines `showDeath()` / `showVictory()`.

### BH-003 - High - `spawnItem` is defined twice; special loot arguments are lost and coordinate-only spawns produce `NaN`

Files:

- `src/js/data.js:75`
- `src/js/items.js:2`
- `src/js/movement.js:52`
- `src/js/map.js:265`

Reproduction:

Run this in the browser console after starting a run:

```js
spawnItem({ x: 10, y: 10 }, null, false);
G.items.at(-1);
```

Observed:

- The spawned item has `x: NaN` and `y: NaN`.
- Reproduced in both `src/index.html` and `dungeon.html`.

Suspected root cause:

- `data.js` defines `spawnItem(r, itemFilter = null, forceHighTier = false)`.
- `items.js` later defines `spawnItem(r)` with the same global name, overwriting the richer implementation.
- `movement.js` calls `spawnItem({x:nx, y:ny}, null, false)` during the low-probability trap-disarm hidden-item branch, but the active `items.js` function expects room dimensions (`r.w`, `r.h`) and computes `NaN`.
- `map.js` calls `spawnItem(r, itemFilter, forceHighTier)` for armory/crypt/treasure/secret room loot, but the active function ignores `itemFilter` and `forceHighTier`, so special room loot tuning is likely not working as designed.

### BH-004 - Medium - Headless balance crashes on shrine interactions

Files:

- `skills/headless-balance/scripts/headless_balance.js:20`
- `src/js/items.js:287`
- `src/js/ui.js:102`

Reproduction:

```bash
node skills/headless-balance/scripts/headless_balance.js --classes warrior --per-class 20 --seed-base 7000 --max-turns 5000 --output tmp.json
```

Observed:

- Many runs end as `fatal_script_error` with `showShrinePrompt is not defined`.
- With 20 runs per class at seed base 7000:
  - Warrior: 16 errors
  - Rogue: 10 errors
  - Mage: 17 errors
  - Paladin: 16 errors
  - Ranger: 11 errors
  - Barbarian: 17 errors
  - Necromancer: 17 errors
  - Monk: 9 errors

Suspected root cause:

- The headless balance `SOURCE_FILES` list loads `items.js`, where shrine pickup calls `showShrinePrompt(it)`.
- The runner does not load `src/js/ui.js` and does not stub `showShrinePrompt`.

Impact:

- Headless balance results are unreliable for seeds that encounter shrines.

### BH-005 - Medium - Mobile landscape layout wastes a vertical column on the log strip

Files:

- `src/css/style.css:96`
- `src/css/style.css:116`

Reproduction:

1. Open the game in a mobile landscape viewport, e.g. `844x390`.
2. Start any class.

Observed:

- The game switches `#game-screen` to row layout.
- `#bottom-bar` remains a flex child between `#map-area` and `#dpad-area`.
- Browser metrics from `844x390`:
  - `#hud`: `left 0`, `right 120`, full height.
  - `#map-area`: `left 120`, `right 545.8`.
  - `#bottom-bar`: `left 545.8`, `right 694`, width `148.2`, height `24`.
  - `#dpad-area`: `left 694`, `right 844`, full height.

Suspected root cause:

- Landscape layout changes the main flex direction to `row` but does not reposition or hide the portrait log bar as a landscape-aware panel/overlay.

### BH-006 - Medium - Mage Blink can spend a turn and cooldown without moving

Files:

- `src/js/combat.js:527`
- `src/js/combat.js:534`

Reproduction:

Run this after starting a Mage run:

```js
initGame('mage');
G.player.lvl = 5;
G.ability2Cooldown = 0;
G.map = Array.from({ length: MAP_H }, () => Array(MAP_W).fill(TILE.WALL));
G.player.x = 10;
G.player.y = 10;
G.map[10][10] = TILE.FLOOR;
G.visible = new Set([10 * MAP_W + 10]);
G.seen = new Set(G.visible);
G.enemies = [];
doAbility2();
```

Observed:

- Position remains `10,10`.
- Turn advances from `0` to `1`.
- Cooldown is spent (`7` after immediate turn decrement).
- Log says `Blinked to a safe location!`.
- Reproduced in both `src/index.html` and `dungeon.html`.

Suspected root cause:

- Blink candidate list includes the player's current tile and only excludes enemies.
- It also does not exclude trap tiles, though landing on a trap was not fully investigated in this pass.
