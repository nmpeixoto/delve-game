# Mobile Shop Scroll Tap Guard Design

## Problem

On mobile, shop buy and sell rows call `buyItem()` or `sellItem()` from `ontouchend`. When a player drags inside the scrollable shop modal, the touch can still end on a row and trigger a transaction. This makes scrolling the shop risky, especially in the sell tab where accidental sells are destructive.

## Goals

- Preserve single-tap buy and sell for intentional taps.
- Ignore touch gestures that moved enough to be a scroll.
- Ignore touches where the shop modal scroll position changed.
- Keep desktop mouse clicks working as they do today.
- Keep the change scoped to `src/` development files, then rebuild `dungeon.html`.

## Non-Goals

- No two-step confirmation flow.
- No new framework, build tool, package, asset, or service worker change.
- No redesign of shop inventory grouping or prices.

## Approach

Add a shared shop touch guard in `src/js/shop.js`. The guard records the starting touch coordinates and `#shop-modal.scrollTop` on `touchstart`, then validates `touchend`. A touch counts as a tap only if movement stays below a small threshold and the modal did not scroll. Buy and sell rows will keep `onclick` for desktop and use the guard for mobile `ontouchstart`/`ontouchend`.

Update shop row CSS in `src/css/style.css` so `.shop-item` and `.sell-item` use `touch-action: pan-y`. This tells mobile browsers that vertical gestures on these rows should scroll naturally while still allowing tap handling.

## Components

- `src/js/shop.js`
  - Add touch state helpers for shop rows.
  - Change generated buy rows to use guarded touch handlers.
  - Change generated sell rows to use guarded touch handlers.

- `src/css/style.css`
  - Change `.shop-item` and `.sell-item` touch behavior from `manipulation` to `pan-y`.

- `dungeon.html`
  - Regenerated by `npm run build` after source edits.

## Data Flow

1. Player touches a shop row.
2. Guard stores touch start position and modal scroll position.
3. Player lifts their finger.
4. Guard compares finger movement and scroll delta.
5. If valid, the transaction runs once.
6. If movement or scroll occurred, the transaction is ignored.

## Edge Cases

- Multi-touch gestures are ignored.
- If the modal is missing, the guard falls back to treating scroll delta as zero.
- A valid tap still prevents the follow-up synthetic click to avoid double transactions.
- Desktop click behavior remains unchanged.

## Testing

- Run `npm test` for logic regression coverage.
- Run `npm run build` to regenerate `dungeon.html`.
- Run `node test.js` with a local static server for browser smoke coverage of `src/index.html` and `dungeon.html`.
- Manually verify mobile-sized shop behavior if a browser session is available: dragging the shop list must not buy or sell, while a still single tap must.
